Levit LLM Guidance
==================

Use this document when building, refactoring, reviewing, or explaining Dart and
Flutter code with Levit.

For routine work, start with LLM-Short.txt. Read this file when you need full
detail on scopes, stores, async DI, middleware, or unfamiliar APIs. See
AGENTS.md for the agent entry point.

Levit is an ownership-first runtime, not only a state-management package. Its
core model is:

1. A scope defines who can resolve what.
2. A controller owns cleanup for a unit of logic.
3. A reactive value defines what can trigger recomputation or rebuild.
4. Flutter bindings attach those rules to widget lifecycles.


Repo Map
--------

```
packages/core/       levit_scope, levit_reactive, levit_dart_core,
                     levit_flutter_core, levit_monitor
packages/kits/       levit, levit_dart, levit_flutter
examples/            runnable reference apps
```

Canonical examples:

- examples/task_board        — controllers, reactive collections, task patterns
- examples/route_scopes      — LRouteScope, LAsyncRouteScope, LScope.put
- examples/scope_playground  — scope boundaries and DI exploration
- examples/async_catalog     — async dependencies and LAsyncView patterns
- examples/nexus_studio      — full multi-package app (app + server + shared)


Package Choice
--------------

Prefer the high-level kits unless there is a specific reason to use core
packages.

Layering (top to bottom):

```
levit_flutter  →  levit_flutter_core + levit_dart (+ Flutter kit widgets/mixins)
levit          →  levit_dart
levit_dart     →  levit_dart_core (+ task/loop/time/selection mixins)
levit_dart_core → levit_scope + levit_reactive
```

Imports:

- Flutter app code: package:levit_flutter/levit_flutter.dart
- Pure Dart app, CLI, server, or shared domain: package:levit/levit.dart
- Dart utility layer only (tasks, loops, mixins): package:levit_dart/levit_dart.dart
- Flutter bindings only: package:levit_flutter_core/levit_flutter_core.dart
- Dart composition APIs only: package:levit_dart_core/levit_dart_core.dart
- Reactive primitives only: package:levit_reactive/levit_reactive.dart
- DI/scope only: package:levit_scope/levit_scope.dart
- Runtime telemetry: add levit_monitor separately

Do not add levit_monitor unless diagnostics, event export, redaction, or
shadow-state debugging is required.


Bootstrap
---------

In Flutter apps, call Levit.enableAutoLinking() in main() before runApp:

```dart
void main() {
  Levit.enableAutoLinking();
  runApp(const MyApp());
}
```

Auto-linking registers reactive values created during controller construction
or onInit with the controller lifecycle. With it enabled, field-level reactives
on a LevitController are often cleaned up automatically.

Still call autoDispose(...) explicitly for:

- Non-reactive resources (TextEditingController, streams, timers).
- Reactives created outside controller init (e.g. dynamically added items).
- Workers, computed values, and objects not covered by auto-linking.

levit_monitor enables auto-linking when initialized.


Dependency Model
----------------

Use the Levit facade for application-level dependency registration and lookup.

Registration and resolution:

- Levit.put(() => ...) — eagerly creates and registers (builder runs immediately).
- Levit.lazyPut(() => ...) — creates on first resolve.
- Levit.lazyPutAsync(() async => ...) — async lazy dependency; resolve with findAsync.
- Levit.find<T>() — resolve from the current active scope.
- Levit.findOrNull<T>() — resolve or null.
- Levit.findAsync<T>() / Levit.findOrNullAsync<T>() — async resolution.
- Levit.isRegistered<T>() / Levit.isInstantiated<T>() — registration probes.
- Levit.delete<T>() / Levit.reset() — remove or clear scope contents.

Use tag to disambiguate multiple registrations of the same type.
Use permanent: true for dependencies that should survive scope reset.
Use isFactory: true on lazyPut when each resolution should create a new instance.

Register global or app-wide services in the app/root scope. Register feature,
route, or screen dependencies in the scope that owns that lifetime.

LevitStore and LevitAsyncStore
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Use stores for portable, reusable state definitions (a "recipe" passed by
reference, instantiated per scope):

```dart
final counterStore = LevitStore<int>((ref) {
  final count = ref.autoDispose(0.lx);
  return count();
});

// In Flutter:
LScopedView.store(counterStore, builder: (context, count) => ...);
context.levit.find<int>(key: counterStore);
```

LevitAsyncStore resolves async setup via LevitStore.async(...).

Flutter scope and context APIs
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Prefer Levit widgets over manual scope handling:

- LScope, LAsyncScope — subtree-owned lifetimes.
- LScope.put — create a scope and eagerly register a root dependency.
- LScopedView, LScopedAsyncView — create a child scope and consume a dependency.
- LRouteScope, LAsyncRouteScope — route identity as the ownership boundary.
- LScope.capture — re-provide the nearest scope to dialogs and overlays.

Consume dependencies from widget code via context.levit (not bare Levit.find
when an LScope is in the tree):

- context.levit.find<T>(), findOrNull, findAsync, findOrNullAsync
- context.levit.put, lazyPut, lazyPutAsync, putOrFind
- context.levit.isRegistered, isInstantiated
- context.levit.find<T>(key: store) for LevitStore / LevitAsyncStore

Use LView / LAsyncView when the dependency already exists in the nearest scope.

Do not manually create, pass, or dispose scopes in normal widget code.
Manipulate scopes directly only for advanced cases: route/overlay bridging,
non-widget Dart integration, explicit test scopes, or framework-level work.


Design Workflow
---------------

Before writing code:

1. Identify the lifetime boundary: app, route, feature, subtree, request, task,
   or test.
2. Create or reuse the matching Levit scope.
3. Put dependencies/controllers in the scope that owns them.
4. Keep reactive state near the controller/store that owns the mutation rules.
5. Use focused Flutter rebuild widgets around only the UI that reads reactive
   values.
6. Add cleanup explicitly for long-lived resources.


Reactive State
--------------

Use .lx for reactive values. Both call syntax and .value are valid on LxVar:

```dart
final count = 0.lx;

count();       // read
count(1);      // write
count.value;   // read
count.value++; // write
```

Use .named('label') for diagnostics and monitoring:

```dart
final query = ''.lx.named('query');
```

Collections:

```dart
final tasks = LxList<TaskItem>().named('tasks');
final selected = <String>{}.lx;
final cache = <String, int>{}.lx;
```

Use computed values for derived state:

```dart
final doubled = LxComputed(() => count() * 2);
final remote = LxComputed.async(() async => await fetchCount());
```

Use LxStatus<T> for async operation state (LxIdle, LxWaiting, LxSuccess,
LxError). Render with LStatusBuilder.

Use workers for side effects:

```dart
final worker = LxWorker(count, (value) {
  print('count=$value');
});
```

Close long-lived reactive objects, computed values, and workers when they are
not owned by a controller, store, or scope that handles cleanup.

Use Levit.batch or Levit.batchAsync (preferred in app code) when multiple
mutations should propagate as one update.


Controllers
-----------

Use LevitController for feature or screen logic.

```dart
class CounterController extends LevitController {
  final count = 0.lx.named('count');

  // Field-level autoDispose works for non-reactive resources:
  late final addController = autoDispose(TextEditingController());

  void increment() => count(count() + 1);

  @override
  void onInit() {
    super.onInit();
    // Explicit when auto-linking is off or for reactives created in onInit:
    autoDispose(count);
  }
}
```

Prefer controllers when logic owns resources, subscriptions, workers, async
tasks, or multiple reactive values.

Keep widget code thin. Put mutation rules and derived state in controllers or
stores.


Flutter Patterns
----------------

App root with a shared controller:

```dart
runApp(
  LScope.put(
    () => AppController(),
    child: const MyApp(),
  ),
);
```

Screen-owned controller:

```dart
return LScopedView<CounterController>.put(
  () => CounterController(),
  builder: (context, controller) => LWatch(
    () => Text('Count: ${controller.count()}'),
  ),
);
```

Widget guide:

- LWatch — build function reads one or more reactive values implicitly.
- LBuilder — observe one explicit LxReactive.
- LSelectorBuilder — small local derived values in hot UI paths.
- LStatusBuilder — LxStatus<T> waiting/error/success rendering.
- LView — dependency already exists in the nearest scope.
- LAsyncView — async dependency resolution with loading/error builders.
- LScopedView / LScopedAsyncView — create a child scope and own the dependency.
- LScope / LAsyncScope — subtree-owned lifetimes.
- LRouteScope / LAsyncRouteScope — route identity as the ownership boundary.
- LScope.capture — dialogs and overlays that must keep a page-local scope.

levit_flutter kit additions (import levit_flutter):

- LKeepAlive — preserve expensive subtrees in scrollable lists.
- LListItemMonitor — subtree lifecycle monitoring.
- LevitAppLifecycleMixin — react to app pause/resume/detach.
- LevitLoopExecutionLifecycleMixin — pause/resume loop execution with app lifecycle.

Avoid manual scope plumbing in ordinary Flutter widgets. Prefer Levit
scope/view widgets, and only pass scopes explicitly when crossing route,
overlay, dialog, or integration boundaries.


Pure Dart Patterns
------------------

Prefer Levit.runInScope for tests and short-lived workflows:

```dart
final result = Levit.runInScope(() {
  Levit.put(() => ApiClient());
  return Levit.find<ApiClient>().ping();
});
```

For long-lived ownership, use Levit.createScope explicitly:

```dart
final scope = Levit.createScope('app');

scope.run(() {
  Levit.put(() => ApiClient());
  final count = Levit.put(() => 0.lx, tag: 'count');
});

scope.dispose();
```

Dispose scopes explicitly. Child scopes should own feature-specific overrides
and clean them up when the feature ends.


Task Utilities
--------------

Use levit_dart utilities when controller-owned work needs scheduling, retries,
cancellation, caching, progress, or reactive busy state.

Task mixins:

- LevitTasksMixin — scheduling, retries, caching, cancellation without
  UI-facing reactive task state (tasksEngine.schedule(...)).
- LevitReactiveTasksMixin — reactive task details, busy state, and progress
  observable by UI or runtime code (runTask(...), tasks, isBusy, totalProgress).

Other levit_dart mixins:

- LevitTimeMixin — debounce, throttle, periodic timers tied to controller life.
- LevitSelectionMixin<T> — single/multi selection state.
- LevitLoopExecutionMixin — managed periodic/continuous workloads.


Middleware
----------

Levit has two middleware channels. Prefer the Levit facade in application code.

Dependency injection middleware (resolve/register/dispose events):

```dart
const auditToken = #di_audit;

Levit.addDependencyMiddleware(AuditMiddleware(), token: auditToken);
Levit.removeDependencyMiddlewareByToken(auditToken);
```

Reactive/state middleware (mutation and lifecycle hooks):

```dart
const traceToken = #state_trace;

Levit.addStateMiddleware(TraceMiddleware(), token: traceToken);
Levit.removeStateMiddlewareByToken(traceToken);
```

Low-level equivalents: LevitScope.addMiddleware / Lx.addMiddleware.

Use one stable token per concern. Re-registering with the same token replaces
behavior in place.


Anti-Patterns
-------------

Avoid:

- Registering feature-owned state in a global/app scope.
- Letting controllers or reactive values outlive their owning scope.
- Using Levit.find in widget code when context.levit.find is required.
- Rebuilding large Flutter subtrees for small reactive reads.
- Putting business mutation rules directly in widgets.
- Manually creating or passing scopes in ordinary Flutter UI code.
- Using monitoring as normal application logic.
- Mixing unrelated lifecycle models when Levit can own the boundary.
- Using low-level core packages in app code when levit or levit_flutter is
  enough.


Validation
----------

For repo changes, run the narrowest relevant check first, then broaden if
needed.

